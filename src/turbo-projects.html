Turbo-geth (Silkworm) projects

Header and Block downloading

Turbo-geth currently using the machinery for downloading headers and blocks that it has inherited from go-ethereum (It does not download receipts and always regenerates them itself at the moment). This machinery consists of downloader, block_fetcher, and filter, and it has been built for serving the sync mode where all the sync stages are happening concurrently. In order to get to the alpha release quicker, we “hacked” this machinery and plugged it into the beginning of our “staged” sync. But there is an architectural impendance there, and the downloader/block_fetcher/filter combo looks like an overkill for the staged sync. Therefore, we are replacing this machinery with a more straightforward, less concurrent one, with hopefully better documentation in the end. You can get some ideas from this document: https://github.com/ledgerwatch/turbo-geth/blob/master/docs/programmers_guide/header_download.md. There are a few functional differences of this new machinery from the existing one:

  1. It will download headers “backwards”, i.e. from the most recent block height toward less recent.
  2. It will include hard-coded “anchors” throughout the history of the chain, to enable downloading from many peers at the same time. Current implementation has something similar, called skeleton, where it loads headers with the gap of 192. In the new machinery, skeleton will be hard-coded, and not dynamically loaded.
  3. It will not try to select a “leading” peer and try to download all the headers from it. Instead, it will defer the peer management to another component that will be separated later.

Separation of consensus engine

Currently there are two consensus engines embedded in the turbo-geth code (inherited from go-ethereum): EtHash PoW and Clique PoA. The layer of abstraction is an interface Engine in the consensus/consensus.go. From our point of view, the interface is already too specific. Adding a new consensus engine (like AuRa or BFT variants), or modifying an existing one would require the modification of the “core” of turbo-geth, and it will have to be implemented on Go language. In order to make it more flexible, we are separating consensus engine into a separate process that will communicate with the turbo-geth process (Core) via gRPC (we already use gRPC for communication between RPC daemon and the Core).
First of all, we are considering two types of instantiations of any consensus engine:

  1. Verifier
  2. Sealer

Currently the interface we are experimenting with consists of two asyncronous streams, going in opposite directions: Core => Engine, and Engine => Core. The stream Core => Engine would carry these types of messages:

  1. Verify this header (header is part of the message)
  2. Here is the header you asked for
  3. Seal this header template (header template is part of the message)

The stream Engine => Core would carry these type of messages:

  1. Header with hash X is valid
  2. Header with hash X is invalid
  3. Please send me header with hash X + N of its ancestors
  4. Here is the sealed header that you sent me template for

As mentioned before, the current idea is to make the interface asynchronous, instead of request/response, though we might be able to transform one into another. We believe that this interface will be sufficient to implement EtHash PoW and Clique PoA. But it is clear to us that AuRa PoA cannot be supported by this interface, because AuRa engine requires access to the state of certain smart contracts. Preliminary extension to the interface (pending better understanding of AuRa PoA mechanism) is to add an extra message type in each direction. The stream Engine => Core gets the message type:

  1. Please send me the state of smart contract with address A after block hash X or any of its ancestors (whatever is available)

The stream Core => Engine would need another message type to send the state of a smart contract to the Engine. It may be the the state of the requested smart contract is not available for the block hash X, but it is available for some ancestors. In that case, the Core would send the latest state it has, the Engine would be able to verify the header following that state, report it to the Core and ask for the state after the next block. The protocol quickly becomes quite tricky, and we’ll need to spend some time to create a minimal interface to deal with the complexity. One observation from this initial design is that AuRa is much harder to implement than Clique.

Block headers and block bodies (and later state snapshot) on BitTorrent

Instead of downloading historical headers and block bodies via eth protocol, we can seed and download them via BitTorrent. Later on, we can download state snapshots too.
Important consideration here is that the downloaded files are kept in read-only mode, so that by default, they are getting “seeded” to other BitTorrent peers. In order to make it work, we would need to support some form of Database overlay mechanism.

DRI Andrea

Transaction replay stage in C++ (using evm one)

We trying using evm one before, but there was a large overhead because of the interfacing between Go and native code. The new component wraps evm one into the extra logic that implements intra-block-state with rules for cross-contract invocations, self-destructing, refunds.

DRI Andrey, Greg

Non-bloom based indices for Logs/Receipts

To achieve acceptable performance of eth_getLogs JSON RPC calls.

If we make the index of logs “precise”, meaning that:

  1. the “value” in the index is not just block number, but ``{block_num; tx_index; log_index}`
  2. index does not have false positives or false negatives

then we can remove most of the information from the stored receipts/log, i.e. remove all the address and topic fields, and only leave data fields. Or perhaps we can just attach that data field to the index entry.

If we end up having an index with embedded data fields, which is smaller than the current receipts bucket (I suspect this is going to be the case), then we can probably stop storing receipts by default (we can recompute them from history + history index), but instead we build up log index at the stage5 (transaction replay) using ETL

DRI Igor

Custom sync state for state analytics

Currently, the state analytics scripts are working either on the chaindata file, or via gRPC interface. If we build them as custom stages instead, this has some advantages:

  1. We do not have a problem of long running transactions inflating LMDB database due to MVCC
  2. Performance is likely to be better because of the local access to the database
  3. There is need to solve consistency issues - the database will be consistent to the last performed cycle.

Pesistence of transactions in the transaction pool

Currently persistence of transactions into a file transactions.rlp does not work.
We would like to not just fix it, but make the transaction persistence more powerful, so that the transactions in the pool can be accessed by the RPC daemon (to suport requests about pending transaction) and by research tools.

Python library to work with data gather by turbo-geth and retrace transactions using Silkworm components

Python is a very popular tool for researchers. We would like to create a python library that would connect to the Core or turbo-geth via gRPC to be able to explore the data, and re-trace transactions from the past (and this is where we would need Silkworm’s execution component. In a way, this is an alternative to JSON RPC, which provides a more convinient intergration and perhaps better performance.
